- **Refactor the `relations` edge table into "real" SQL relations (typed FK columns /
  per-kind join tables)**: assessed 2026-07-03 (commit `637e3a9`), rejected — the premise is
  wrong. The table already IS physical SQL: `subject_id`/`object_id` are enforced FKs to
  `records` (`PRAGMA foreign_keys = ON`, `canonical-db.ts:66`), STRICT, CHECK on provenance,
  indexed both directions (`schema-ddl.ts:88-92`). What it isn't is *typed per kind*, and it
  can't be without data loss: (a) even the big attachment kinds are N:M in the live data
  (has_metric: 82 objects with >1 subject; has_timeline_event: 142; has_treatment: 113), so
  FK columns on child tables can't represent them; (b) subject kinds are mixed per relation
  kind (has_metric subjects: entity 4,509 / route 1,001 / project 334 / corridor 126), so a
  typed column would just be a generic FK to `records` — which is what exists; (c) every one
  of the 20,640 edges is itself a record carrying evidence_refs (20,640/20,640),
  assertion_status, as_of_date (18,042), provenance, canonicalize_decision_id — plain FK
  columns can't carry this, and per-family join tables would replicate ~8 metadata columns ×
  17 tables and fork the ingest/dedup/canonicalize pipeline that treats relations as records;
  (d) the vocabulary is open by design (~600 raw kinds, ~350 with count ≤2) with the closed
  17-value `relation_family` layer as the documented answer
  (`docs/data-driven-ontology-decisions.md`, `docs/sqlite-migration-plan.md:36` — "Edges are
  already FK-clean … loses nothing"). The refactor's would-be benefits already exist:
  integrity (FKs on), typed query surfaces (the 11 canonical views), endpoint-kind typing
  (`relationEndpointShapeIssue` in validate). The 706 `duplicate_relations` triples are the
  temporal assertion model (same edge, different source/as_of_date — `resolved_status`
  depends on it), not dirt. One legitimate residue: the `relation_family` CHECK
  (`docs/sqlite-migration-plan.md:132` deferred it "until saturated") — queued as **plan 016**
  by owner request the same day, after finding that the schema-audit saturation gate can
  structurally never fire for it (18 distinct values > `ENUM_MAX_DISTINCT = 12`).
