# Immutable MTA LLM Wiki Spec

DO NOT EDIT THIS FILE during ordinary agent work.

This document is the stable project charter. It should be read as a north-star spec, not as a living
implementation plan. Agents may update working specs, prompts, schemas, code, tests, logs, and run
notes, but they must not rewrite this file unless the project owner explicitly asks to edit
`docs/immutable-mta-llm-wiki-spec.md` in the current conversation.

## Purpose

The MTA LLM Wiki exists to build durable, source-backed knowledge about MTA, NYC DOT, and related
public transit documents, especially around bus reliability, bus priority, route changes, corridor
projects, service delivery, capital work, operations, public claims, and evaluation evidence.

The project is not a generic chat-over-documents system. The goal is to turn public and official
sources into reusable knowledge that future agents, humans, scripts, and downstream products can
inspect, query, validate, and extend.

The wiki should answer hard transit-product questions by accumulating what has already been learned:
which projects exist, which routes or corridors they affect, what agencies said, what metrics were
claimed, what treatments were proposed or implemented, what evidence supports those claims, and what
remains uncertain.

## Core Idea

The central design is a filesystem-first LLM wiki with two complementary knowledge layers:

1. Structured, programmatically accessible records.
2. Human-written wiki context that adds synthesis and durable judgment.

Structured data is the canonical machine layer. It stores source-backed observations, entities,
relationships, dates, metrics, claims, statuses, caveats, and evidence handles in forms that programs
can later consume.

Wiki prose is the complementary interpretation layer. It should explain what the structured data
means, connect related records, preserve useful context, identify caveats, and record questions that
do not yet fit the schema.

Neither layer replaces the other. If the system only stores prose, it becomes hard to query and
validate. If the system only stores structured records, it loses the accumulated human context that
makes a wiki useful.

## Pipeline Philosophy

The pipeline is intentionally split into stages.

The first agent reads one source at a time and submits structured observations. It is an extractor
and router, not the final author of pages. It should preserve source literals where they matter,
classify only bounded fields, submit uncertainty honestly, and attach evidence handles where the
record depends on the source.

The deterministic runner stores submissions, validates them, assigns ids, materializes canonical
records, and creates or updates wiki pages. Page generation is deterministic from submissions and
canonical data.

The second agent reads the materialized wiki and writes useful Markdown context inside writer-owned
regions. It should add value that generated frontmatter cannot: synthesis, caveats, relationships,
identity notes, public-safety warnings, unresolved questions, and observations that may later become
structured fields.

This separation is a central design choice. The writer must not become the owner of canonical facts,
and the extractor must not become the owner of wiki prose.

## Data Philosophy

The data model should distinguish open-world discoveries from closed-world classifications.

Open-world objects are discovered from sources. These include projects, interventions, corridors,
routes or route mentions, timeline events, official claims, metric claims, source gaps, named
programs, studies, reports, public bodies, and related entities. The system should be willing to
store them before it knows whether they are globally canonical.

Closed-world classifications are bounded labels used to make data queryable. These include treatment
kind, event kind, date role, status, metric family, relation kind, truth status, review state, and
publication posture. These enums should evolve carefully and remain permissive early.

Source literals should be preserved. The system should not normalize away the raw words that a source
used for a project name, date, metric value, corridor, restriction, quote, table cell, or caveat.
Normalized companion fields may be added for queryability, but they should not erase the original
source wording.

## Evidence Philosophy

Evidence exists for provenance, validation, audit, and later rehydration. It is not primarily a prose
format.

Agents should normally submit evidence handles rather than long copied excerpts. A useful evidence
handle points to a stable source, page, block, range, table, or other citeable location. The runner
owns validation, hashes, and any later display of snippets.

Citation precision should be progressive. Expensive LLM effort should be focused where precision
matters most: metrics, dates, costs, status claims, launch or completion claims, direct official
claims, contradictions, and other publication-sensitive facts. Lower-risk identity or scope records
can begin with coarser evidence if marked honestly and refined later.

Raw text still matters when the text itself is the data. Project names, official wording, raw dates,
metric values, quotes, and table cells should be stored as source literals in structured fields, with
evidence handles attached.

## Page Philosophy

Generated wiki pages should start as YAML frontmatter plus explicit writer markers. The materializer
owns generated frontmatter. The writer owns only the content inside writer regions.

Programmatic Markdown bodies should be avoided unless they are clearly generated indexes or reports.
For entity pages, source pages, project pages, route pages, and corridor pages, generated structure
belongs in frontmatter or canonical JSONL. Human-readable Markdown should be written by the writer
agent only when it adds durable value.

Pages may be created immediately from structured submissions. This is useful because source-backed
entities become inspectable and linkable as soon as they are discovered. Page creation must be
idempotent and resumable.

## Identity And Ids

Ids should be stable, compact, and cheap for agents to read and repeat.

Agents should not submit hashes. Hashes, deterministic ids, collision handling, and provenance
bookkeeping are runner responsibilities. Page filenames may be descriptive for humans, but they
should not include long hash suffixes unless a collision requires one.

Upstream source ids and provenance should still be preserved in metadata. Compact local ids are an
ergonomic layer for this wiki, not a denial of source history.

## Source Philosophy

The v0 input contract is filesystem-first. Source artifacts can be copied into this project or a
sandbox workspace so agents can read them reliably. More sophisticated source adapters, dedupe
strategies, and non-duplicating storage can come later.

Each run should process one source or one bounded source window at a time. The system can scale to
large corpora through resumable staging, append-only submissions, deterministic materialization,
validation, and later refinement passes.

Transcripts and run logs are audit surfaces, not source evidence. Agents should not use transcripts
as factual inputs for extraction or writing.

## Desired Outcome

The successful version of this project is a growing wiki that can be trusted more than an ad hoc LLM
chat because it remembers its work, preserves provenance, separates data from prose, and makes
uncertainty visible.

It should help future agents and programs answer questions such as:

- What projects, interventions, and corridor efforts exist?
- Which routes, places, districts, or communities are affected?
- What did official sources claim, promise, evaluate, or caveat?
- What timeline events happened or were proposed?
- What metrics, costs, ridership figures, and service-delivery claims were stated?
- What evidence supports each observation?
- What is unresolved, contradicted, missing, or only source-stated?

The project should optimize for durable accumulated knowledge over one-off summaries. Every run
should make the wiki more useful, more auditable, or easier to extend.
